# 跨端设计：以捷安特服务端为事实源的多端去重

> 日期：2026-07-02
> 状态：已确认
> 适用范围：所有端（desktop / android / 未来 windows、ios）

## 1. 背景与问题

本项目为多端项目，多个端（桌面 Java 版、Android 版，未来可能有 Windows、iOS 版）**可能同时运行**。原设计中每个端用自己的本地数据库（`fit_url` 唯一键）做去重，端与端之间数据不互通，会导致同一文件被不同端重复下载、重复上传。

## 2. 方案：捷安特服务端作为唯一事实源

不引入统一数据源（无需自建服务端/云数据库）。利用捷安特[「已上传文件列表」接口](../api/giant.md#3-已上传文件列表全量多端去重的事实源)（`all_upload`），**以「捷安特账号下已有哪些文件」这一天然全局一致的状态作为去重依据**：

- **匹配键**：`all_upload` 返回的 `file`（上传时原始文件名）＝ 同步工具上传的文件名 ＝ 顽鹿 `fitUrl`（全局唯一）；
- **去重规则**：顽鹿活动的 `fitUrl` 在服务端列表中出现过（**任意状态**）→ 跳过，不再下载/上传；
- **同名多条记录**（服务端允许重复上传）：任意一条 `status=成功` 即视为同步成功；全部非成功 → 视为「已上传但处理失败」；
- **各端本地数据库降级为展示层**：只记录本机的同步会话与操作历史，供界面展示与排查，**不再承担去重职责**。各端记各自的账，互不感知。

## 3. 状态语义升级

`all_upload` 的 `status` 是服务端**真实处理状态**，比「上传接口返回成功」更进一步：

| 服务端观察结果 | 含义 | 各端行为 |
|---|---|---|
| 文件名无记录 | 未同步 | 正常下载 + 上传 |
| 任意一条记录成功 | 真正同步完成 | 跳过；本机记录 reconcile 为已同步 |
| 有记录但全部非成功 | 已上传但处理失败 | 跳过（不自动重传）；界面**标红**提示，人工处理 |

**失败重试规则的变化**：原「本地库有任意记录（含失败态）即永久跳过」规则废弃。下载失败/上传失败的文件在服务端没有记录，下次同步会**自然自动重试**（任何一端都可能完成它）；只有「已上传但处理失败」不自动重试（重传大概率无效），标红人工处理。

## 4. 效率约定（all_upload 无分页，可达数千条）

- **每次同步会话只调用一次** `all_upload`，结果在内存构建 `HashSet<文件名>`（数千条字符串内存占用可忽略，成员判定 O(1)）；
- 调用频率＝同步频率（如每 6 小时一次 + 少量手动触发），每天个位数次请求，对服务端友好；
- **不做本地缓存替代服务端查询**——缓存会重新引入多端不一致，违背事实源原则；
- 未来记录量增长到数万条时的备选优化（当前不做）：按 `time` 字段截断近期窗口 + 本地维护「历史已确认集合」增量合并。

## 5. 已知限制（已确认接受）

1. **并发窗口**：两端恰好同时同步时都查到「服务端没有」，可能同时上传同一文件。同步间隔为小时级、上传为秒级，概率极低，且不影响数据正确性（服务端允许同名多条记录）。暂无低成本解法，接受。
2. **非本工具上传的记录无法匹配**：同一场骑行若经其他途径（如码表直连）上传到捷安特，文件名不同，无法识别为重复。与原设计一致，非本方案引入的问题。
